# MLOPS Lifecycle Structure

This folder groups the project by MLOps lifecycle step so the flow is easy to follow.

## Project Root
- `requirements.txt` installs the notebook, MLflow, Prefect, DVC, and Feast dependencies.
- `FraudClassifier.ipynb` is the beginner-friendly notebook version.

## 01 Data Versioning
- folder: `01_data_versioning_dvc`
- `01_creditcard_dataset.csv.dvc` points to the exact dataset version used for training.
- `01_creditcard_dataset.csv` is the local dataset copy managed by DVC.

## 02 Feature Store
- folder: `02_feature_store_feast`
- `feature_repo/feature_store.yaml` configures the local Feast repo.
- `feature_repo/02_fraud_feature_definitions.py` defines the Feast entity, feature view, and feature service.
- `feature_repo/data/02_fraud_feature_source.parquet` is the generated batch feature source.
- `feature_repo/data/02_latest_online_features.json` captures an example online feature lookup.

## 03 Orchestration
- folder: `03_orchestration_prefect`
- `03_prefect_mlops_pipeline.py` runs the end-to-end flow with Prefect.
- `03_pipeline_params.yaml` stores reproducible pipeline parameters.

## 04 Experiment Tracking
- folder: `04_experiment_tracking_mlflow`
- `04_mlflow.env.example` shows how to configure local or remote MLflow.

## 05 Pipeline Versioning
- folder: `05_pipeline_versioning_dvc`
- `dvc.yaml` defines the reproducible MLOps stage.
- `dvc.lock` stores the exact dependency and output hashes from the latest successful run.

## 06 Model Artifacts
- folder: `06_model_artifacts_joblib`
- This folder stores the best model, scaler, metrics, and metadata generated by the pipeline.

## 07 Model Serving
- folder: `07_model_serving_bentoml`
- `07_export_bentoml_model.py` turns the trained model artifacts into a BentoML bundle.
- `07_bentoml_service.py` exposes a simple prediction API for serving.

## 08 Monitoring
- folder: `08_monitoring_scipy`
- `08_run_monitoring_checks.py` creates a production-like batch, scores it, and checks for drift.
- generated monitoring files show whether incoming data still looks like training data.

## 09 Retraining Trigger
- folder: `09_retraining_trigger_policy`
- `09_decide_retraining.py` reads the monitoring results and decides whether retraining should happen.
- `09_retraining_policy.yaml` stores the simple business rules for that decision.

## 10 CI CD
- folder: `10_ci_cd_github_actions`
- `.github/workflows/10_mlops_ci_cd.yml` automates the lifecycle in GitHub Actions.
- `10_ci_cd_github_actions/README.md` explains how the automation fits into the MLOps flow.

## 11 Deployment Promotion
- folder: `11_deployment_promotion_mlflow`
- `11_promote_model.py` checks the retraining decision and either promotes or blocks the latest MLflow model version.
- `11_promotion_policy.yaml` stores the alias and approval rules for promotion.

## Useful Commands
- `python MLOPS/03_orchestration_prefect/03_prefect_mlops_pipeline.py`
- `dvc repro MLOPS/05_pipeline_versioning_dvc/dvc.yaml:run_full_mlops_cycle`
- `mlflow ui --backend-store-uri sqlite:///MLOPS/04_experiment_tracking_mlflow/04_mlflow_tracking.db`
- `feast ui` from `MLOPS/02_feature_store_feast/feature_repo`
- `python MLOPS/07_model_serving_bentoml/07_export_bentoml_model.py`
- `bentoml serve MLOPS/07_model_serving_bentoml/07_bentoml_service.py:FraudDetectionService`
- `python MLOPS/08_monitoring_scipy/08_run_monitoring_checks.py`
- `python MLOPS/09_retraining_trigger_policy/09_decide_retraining.py`
- `python MLOPS/11_deployment_promotion_mlflow/11_promote_model.py`
